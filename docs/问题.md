
# Sitlys 全面代码审查与功能差距分析报告

## 一、已确认的 Bug 与代码问题

### 🔴 严重问题

#### 1. 会话窗口 Key 使用 30 分钟截断，导致会话切分逻辑与 PRD 不一致

[ingest.go:157](file:///d:/work/sitlys/server/ingest.go#L157) 中 `sessionWindowKey` 使用 `Truncate(30 * time.Minute)` 生成 key，这意味着**同一访客在 10:05 和 10:35 的两次访问会落入不同的 30 分钟窗口**，即使间隔只有 30 分钟，也会创建两个会话。但 PRD 7.5 明确要求"两次事件间隔不超过 30 分钟归为同一会话"。

**实际效果**：如果访客在 10:01 和 10:29 各有一次访问（间隔 28 分钟），它们会被分配到不同的窗口 key（10:00 和 10:30），从而创建两个会话，违反 PRD 规则。反之，如果访客在 10:31 和 10:59 各有一次访问（间隔 28 分钟），它们会共享同一个窗口 key，但 10:59 与 11:01（间隔 2 分钟）却会被分到不同会话。

**修复建议**：应改为基于 `last_seen_at + 30min` 的滚动窗口判断，而非固定截断。

#### 2. `findOrCreateSessionTx` 递归重试可能死循环

[ingest.go:193](file:///d:/work/sitlys/server/ingest.go#L193) 在 `session_key` 唯一冲突时递归调用自身。虽然理论上窗口 key 应该稳定，但在极端并发场景下，多个 goroutine 可能同时为同一访客创建会话，导致反复 unique 冲突 → 递归 → 再冲突。没有递归深度限制。

**修复建议**：增加最大递归次数（如 3 次），超过后返回错误。

#### 3. `detectUserAgent` 中 Edge 检测顺序错误

[app.go:1106](file:///d:/work/sitlys/server/app.go#L1106) 中 `edg` 在 `chrome` 之前检测是正确的，但 `safari` 的检测在 `chrome` 之后——然而 Chrome 的 UA 也包含 "Safari" 字符串（如 `Mozilla/5.0 ... Chrome/120.0 ... Safari/537.36`），所以 Safari 的检测永远不会对 Chrome 用户生效，这是正确的。但**缺少 Opera 的检测**（Opera UA 包含 "OPR/"），Opera 用户会被误识别为 Chrome。

#### 4. 收入事件的 `amount` 为 0 时仍被记为 revenue 类型

[app.go:1088](file:///d:/work/sitlys/server/app.go#L1088) 中 `normalizeEventType` 只要 `payload.Revenue != nil` 就返回 `"revenue"`，即使 `amount = 0`。这会导致 0 金额的收入事件被写入 `agg_revenue_daily`，产生无意义的收入记录。测试中虽然有 `TestNormalizeEventType` 的 "zero amount revenue" 用例验证了这一点，但从业务角度看，0 金额收入事件不应进入收入聚合。

#### 5. `agg_visitor_daily` 不参与清理

[cleanupHistoricalData](file:///d:/work/sitlys/server/app.go#L1519) 清理聚合表时，`aggregateTables` 列表缺少 `agg_visitor_daily`，导致该表数据永远不会被清理，随着时间推移会无限增长。

### 🟡 中等问题

#### 6. 概览页跳出率和平均会话时长仍查 `sessions` 原表

[http.go:1303](file:///d:/work/sitlys/server/http.go#L1303) 中 `handleOverview` 的 `bounce_rate` 和 `avg_session_duration_seconds` 仍然直接查询 `sessions` 表做 `avg()` 聚合，而不是使用 `agg_overview_daily`。当数据量增长后，这会成为性能瓶颈，与 PRD 中"概览页不再依赖实时 count(distinct) 扫描 sessions 表"的优化目标不一致。

#### 7. `handleOverview` 查了两次相同数据

`handleOverview` 先查了一次 `agg_overview_daily` 获取主指标，然后 `loadOverviewCompare` 又查了当前周期的相同数据。应复用第一次查询结果。

#### 8. 像素采集入口缺少域名校验

[http.go](file:///d:/work/sitlys/server/http.go) 中 `handleCollectPixel` 是 GET 请求，只有 `website_id`（通过 pixel → website 映射获取），但没有 URL/hostname 可供校验。任何人只要知道 pixel ID 就可以通过不断请求 `/collect/p/` 来伪造数据。PRD 11.4 要求域名白名单校验，但像素入口绕过了这个限制。

#### 9. 前端 `dateOffset` 默认时间范围是 -29 到 0，但 `to` 包含当天

[use-app-controller.js:34](file:///d:/work/sitlys/web/src/composables/use-app-controller.js#L34) 默认 `from: dateOffset(-29), to: dateOffset(0)`，这意味着"最近 30 天"。但后端查询使用 `between ? and ?`，如果 `to` 是今天，而今天还没结束，那么今天的数据只有部分，与历史完整天对比时会产生偏差。后端没有对 `to` 做天花板截断。

#### 10. 前端 `overviewTrendPeaks` 中 `Math.max(...array)` 对大数组会栈溢出

[use-app-controller.js](file:///d:/work/sitlys/web/src/composables/use-app-controller.js) 中 `Math.max(...state.overviewTrend.map(...))` 使用展开运算符，当日志数据超过约 65536 条时（虽然趋势数据通常不会这么多），会导致 `Maximum call stack size exceeded`。

#### 11. `tracker.js` 的 `sendBeacon` 使用 `application/json` Content-Type 会被部分浏览器丢弃

[tracker.js:19](file:///d:/work/sitlys/server/tracker.js#L19) 中 `navigator.sendBeacon` 使用 `Blob` 设置 `type: "application/json"`。在部分浏览器（特别是较旧版本的 Chrome）中，`sendBeacon` 对非 `text/plain` 等 CORS 安全类型可能不发送或被拦截。不过现代浏览器已修复此问题，影响较小。

#### 12. 公开分享页缺少速率限制

`handlePublicShare` 端点没有任何速率限制，恶意用户可以通过高频请求公开分享 API 来消耗服务器资源。

### 🟢 轻微问题

#### 13. `normalizeWebsiteDomain` 未处理端口号

如果网站域名配置为 `localhost:3000`，域名校验可能无法正确匹配带端口的请求 hostname。

#### 14. `agg_devices_daily` 中空值被 `nullUnknown` 转为 "Unknown"

[ingest.go:282](file:///d:/work/sitlys/server/ingest.go#L282) 中 `nullUnknown` 将空字符串转为 "Unknown"，但这意味着 "Unknown" 作为主键一部分参与聚合，与前端展示时将空值替换为 "(unknown)" 不一致。

#### 15. 前端没有对 API 请求做防抖/节流

快速切换路由或时间范围时，会触发大量并发 API 请求。虽然 `safeRequest` 会吞掉错误，但不会取消之前未完成的请求，可能导致竞态条件（旧请求的响应覆盖新请求的结果）。

## 三、与同类产品（Plausible / Umami / Matomo / PostHog）对比，欠缺的核心功能

### 🔴 高优先级缺失（同类产品标配）

#### 1. **无 IP 地理位置数据库**
当前 `detectGeo` 完全依赖 CDN 代理头（CF-IPCountry 等），如果 Sitlys 部署在没有 Cloudflare 的环境中，地域数据全部为空。Plausible 和 Umami 都内置了 MaxMind GeoLite2 或类似方案。

**建议**：嵌入一个轻量级 GeoIP 数据库（如 MaxMind GeoLite2 或 IP2Location），在采集时根据客户端 IP 直接查询。

#### 2. **无自定义仪表盘/看板**
PRD 明确将"自定义看板编排"列为非目标，但这是 Plausible、Umami、Matomo 的核心功能。用户无法将最关心的指标组合到一个视图中。

**建议**：至少支持"固定概览卡片顺序"或"选择概览页显示的指标"这种轻量级自定义。

#### 3. **无事件属性分析**
当前自定义事件只记录 `name` 和 `data`（metadata），但无法按 `data` 中的字段做聚合分析。例如 `track("purchase", { plan: "pro", category: "SaaS" })` 无法按 `plan` 或 `category` 维度拆分查看。PostHog 和 Matomo 都支持事件属性分析。

**建议**：至少支持对 `data` 中 Top N 属性值的聚合统计。

#### 4. **无转化目标（Goals）**
PRD 将 Goals 列为非目标，但这是 Plausible 和 Umami 的核心功能。用户无法定义"当用户触发某事件或访问某页面时算作转化"并追踪转化率。

**建议**：可以复用漏斗的基础设施，增加"单步骤目标"概念。

#### 5. **无邮件报告**
Plausible、Umami、Matomo 都支持定期发送邮件报告。Sitlys 作为自托管产品，用户可能不会每天登录查看，邮件报告是重要的留存功能。

**建议**：后续版本增加 SMTP 配置和定时邮件报告。

#### 6. **无数据导入能力**
无法从 Google Analytics、Matomo 等平台迁移历史数据。对于想从其他平台切换过来的用户，这是重要障碍。

### 🟡 中等优先级缺失

#### 7. **无 Webhook/通知**
无法在特定条件触发时（如流量异常、收入里程碑）发送通知。PostHog 和 Plausible 都支持 Webhook。

#### 8. **无 A/B 测试或 Feature Flag**
PostHog 的核心功能，但对于 Sitlys 的定位可能过于复杂。可作为远期规划。

#### 9. **无用户画像/用户详情页**
当前只有访客 ID 和会话列表，无法查看单个用户的行为时间线。Umami 有用户详情页。

#### 10. **无世界地图可视化**
PRD 明确提到"世界地图可视化"属于后续功能。当前地域分析只有表格/条形图，缺少直观的地图展示。

#### 11. **无实时 WebSocket 推送**
当前实时数据需要手动点击刷新。Plausible 使用 WebSocket 实时推送更新。

#### 12. **无数据对比功能（自定义时间段对比）**
当前只有"与上一相同长度周期对比"，无法选择两个任意时间段对比。

#### 13. **无 API Token 认证**
当前所有管理 API 依赖 Cookie 会话，无法用于自动化脚本或第三方集成。应支持 Bearer Token 或 API Key 认证。

### 🟢 低优先级缺失

#### 14. **无暗色主题**
同类产品普遍支持。

#### 15. **无多语言内容翻译管理**
前端 UI 支持中英文，但分析数据中的内容（如页面标题、事件名）没有翻译能力。

#### 16. **无数据采样/近似查询**
当数据量极大时，SQLite 查询会变慢。Plausible 使用 ClickHouse 的近似查询来保证性能。

## 四、代码质量与架构建议

### 1. 单文件过大
[http.go](file:///d:/work/sitlys/server/http.go) 超过 2700 行，[app.go](file:///d:/work/sitlys/server/app.go) 超过 1500 行。建议按功能拆分：
- `http_auth.go` - 认证相关 handler
- `http_analytics.go` - 报表查询 handler
- `http_admin.go` - 管理操作 handler
- `ingest.go` - 已拆出，但可以进一步拆分聚合逻辑

### 2. 缺少请求取消机制
前端切换路由时，之前的 API 请求不会被取消。建议在 `apiRequest` 中支持 `AbortController`，在路由切换时取消未完成请求。

### 3. 缺少数据库连接池监控
SQLite 虽然是单写模型，但 `SetMaxOpenConns(8)` 在高并发读取时可能不够。建议增加运行时指标暴露（如 `/metrics` 端点）。

### 4. 缺少结构化日志
当前使用 `log.Printf`，建议改为结构化日志（如 `slog`），便于日志分析和过滤。
