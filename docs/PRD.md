# Sitlys PRD

## 1. 文档目的

本文档定义 `Sitlys` 的产品范围、业务规则、核心实体、页面结构、接口边界、数据采集规则与验收标准。

本文档的目标不是做市场宣传，而是为 AI 与工程实现提供稳定的产品约束。阅读本文档后，应能直接开展以下工作：

- 拆分后端模块
- 设计 SQLite 表结构
- 设计 HTTP API
- 实现前端页面与交互
- 编写埋点采集脚本
- 编写测试与验收用例

本文档优先保证“可实现、可验证、可拆解”，不追求大而全的叙述。

## 2. 产品定义

`Sitlys` 是一个面向自托管场景的轻量级多站点网站分析系统。

系统采用以下固定形态：

- 后端使用 `Go`
- 默认数据库使用 `SQLite`
- 前端使用 `Vue 3` 与 `<script setup>`
- 前端构建产物通过 `Go embed` 嵌入后端二进制
- 最终交付物为单一可执行文件
- 同一进程同时提供管理后台、采集接口、公开分享页面、静态资源服务

产品核心价值如下：

- 安装简单：下载单个二进制即可启动
- 运维简单：不依赖外部数据库、Redis、消息队列
- 使用简单：首次启动后通过 Web UI 完成初始化和管理
- 部署灵活：适用于 Linux、Windows、macOS 的单机或轻量服务器部署

## 3. 目标用户

目标用户包括：

- 个人站长
- 小团队内部项目维护者
- 需要私有化分析系统的开发者
- 希望替代第三方统计工具的自托管用户
- 需要在弱运维、内网或离线环境中使用统计系统的用户

不面向以下场景：

- 大规模多租户 SaaS 平台
- 高吞吐、超大体量日志分析平台
- 高度自定义 BI 平台
- 复杂组织架构与企业身份体系集成平台

## 4. 产品目标

### 4.1 业务目标

系统必须支持完整主链路：

1. 启动服务
2. 首次初始化管理员
3. 创建网站
4. 获取埋点脚本或像素地址
5. 接入被统计站点
6. 采集访问与事件数据
7. 在后台查看统计结果
8. 按角色和站点范围控制可见性
9. 生成公开只读分享链接

### 4.2 交付目标

系统必须满足以下交付要求：

- 单二进制运行
- 默认使用 SQLite 自动建库建表
- 不依赖 Node.js 运行时提供生产页面
- 不依赖 PostgreSQL、MySQL、Redis、ClickHouse
- 可构建 Linux、Windows、macOS 产物
- 管理台和采集接口由同一服务进程提供

## 5. 产品原则

### 5.1 部署优先

任何设计都不得破坏单二进制、自带数据库、开箱即用这三个前提。

### 5.2 主链路优先

优先保证“初始化、建站、接入、采集、分析、授权、分享”主路径完整可用。

### 5.3 范围清晰

需求必须有明确边界，避免将产品演化为通用数据平台。

### 5.4 权限简单但明确

不做多租户和组织体系，但必须支持多用户、多角色、站点级授权。

### 5.5 查询可落地

分析能力必须适配 SQLite，避免引入难以维护的超重查询模型。

## 6. 非目标

以下内容明确不属于本产品范围：

- 多租户隔离
- 团队 / 组织模型
- SSO / OAuth 企业登录
- Session Replay
- 短链统计
- Journey 报表
- 自定义看板编排
- 高级用户分群
- 云订阅与计费逻辑
- Redis / ClickHouse / 外部 OLAP 兼容层

## 7. 术语定义

### 7.1 网站

网站是统计对象。一个系统内可以管理多个网站，每个网站有独立的统计数据、像素、漏斗、分享链接与访问授权范围。

### 7.2 访客

访客是某网站下的唯一访问个体。访客标识由前端脚本或像素请求提供的外部标识映射而来；同一个外部标识在不同网站下视为不同访客。

### 7.3 会话

会话是访客在某网站上的一次连续访问行为集合。系统采用 `30 分钟无活动即切分新会话` 的规则。

### 7.4 页面浏览

页面浏览是一次页面访问记录，通常由埋点脚本自动发送。

### 7.5 自定义事件

自定义事件是业务方主动上报的行为事件，例如 `signup`、`purchase`、`click_cta`。

### 7.6 收入事件

收入事件是带金额与币种信息的自定义事件，用于收入分析。

### 7.7 像素

像素是用于极简接入的采集入口，通常通过一个图片请求完成上报。

### 7.8 分享链接

分享链接是一个公开可访问的只读报表入口，不需要登录，不具备任何管理能力。

- 用户认证
- 用户与站点授权管理
- 网站管理
- 采集脚本分发
- 像素采集
- 单条与批量事件接收

## 9. 角色与权限

系统不做多租户，但必须支持多角色、多权限以及按站点控制数据可见范围。

### 9.1 固定角色

- `super_admin`
- `admin`
- `analyst`
- `viewer`

### 9.2 角色定义

`super_admin`

- 系统初始化后默认创建的首个用户角色
- 可管理全部用户
- 可管理全部网站
- 可查看全部网站数据
- 可分配用户角色与站点权限
- 可访问系统配置

`admin`

- 可管理被授权的网站
- 可查看被授权网站的全部分析数据
- 可管理被授权网站的像素、分享、漏斗
- 不可管理系统级用户与全局配置

`analyst`

- 可查看被授权网站的数据
- 可使用分析能力
- 不可管理用户
- 不可修改系统配置
- 默认不可修改网站基础配置，除非明确授予管理站点能力

`viewer`

- 仅可查看被授权网站的数据
- 不可创建或修改网站、像素、分享、漏斗
- 不可访问用户管理和系统配置

### 9.3 站点权限模型

除 `super_admin` 外，用户是否能访问某个网站，必须由站点权限决定。

站点权限固定为三态：

- `none`
- `view`
- `manage`

含义如下：

- `none`：不可查看该网站任何数据
- `view`：可查看该网站分析数据，但不可修改网站配置
- `manage`：可查看并管理该网站配置、像素、分享、漏斗

实现约束：

- `none` 作为交互态存在，用于表达“未授权”
- 持久化到 `website_permission` 表时，仅保存 `view` 与 `manage`
- 当某用户对某网站为 `none` 时，数据库中不应存在对应授权记录

系统行为必须满足：

- 角色决定能力边界
- 站点权限决定可见范围
- 不允许仅设置角色而不设置站点范围就隐式获得网站访问能力
- `super_admin` 默认拥有全部网站完全权限，不需要单独配置站点权限

### 9.4 后台交互原则

后台中“角色”和“站点授权”不应拆成两个主入口。

管理后台必须满足以下交互原则：

- 左侧导航只保留 `用户管理`
- 不单独提供一个与 `用户管理` 重复的“角色与授权”主页面
- `用户管理` 页面内完成以下操作：
  - 创建用户
  - 编辑用户基础信息
  - 选择固定角色
  - 按网站分配 `none / view / manage`
- 若需要展示角色说明，可作为 `用户管理` 页中的辅助说明卡片，而不是独立工作流

### 9.5 权限约束

- 未登录用户不得访问管理 API
- 未授权用户不得访问未分配站点的数据
- 只读分享链接只能查看分享范围内的数据
- 非 `super_admin` 不得管理系统用户
- 非站点管理员不得修改站点、像素、分享、漏斗

## 10. 功能范围

- 概览

## 11. 数据采集规则

### 11.1 采集入口

系统提供三类采集入口：

- `tracker.js` 自动脚本入口
- 单条事件上报入口
- 批量事件上报入口
- 像素采集入口

### 11.2 事件类型

系统内事件至少分为以下三类：

- `pageview`
- `event`
- `pixel`

其中：

- `pageview` 用于页面访问
- `event` 用于自定义行为
- `pixel` 用于像素方式触发的采集

### 11.3 事件公共字段

采集层应围绕以下字段设计：

- `website`
- `pixel`
- `url`
- `referrer`
- `name`
- `title`
- `hostname`
- `language`
- `screen`
- `timestamp`
- `id`
- `browser`
- `os`
- `device`
- `country`
- `region`
- `city`
- `utm_source`
- `utm_medium`
- `utm_campaign`
- `utm_content`
- `utm_term`
- `data`
- `revenue`

### 11.4 访客识别规则

- 访客识别基于外部访客标识 `id`
- 同一 `website` 下，相同 `id` 视为同一访客
- 不同网站之间不共享访客身份
- 若请求未提供稳定访客标识，系统可降级使用临时标识，但不保证跨访问稳定性

### 11.5 会话切分规则

- 同一网站、同一访客下，若两次事件间隔不超过 `30 分钟`，则归为同一会话
- 超过 `30 分钟` 则创建新会话
- 会话记录需要维护入口页、退出页、最后活跃时间、事件数、页面浏览数

### 11.6 页面浏览规则

- 页面浏览至少记录完整 URL 与路径
- 分析查询默认按 `url_path` 统计热门页面
- 页面标题作为辅助展示字段，不作为主聚合键

### 11.7 来源归因规则

归因基础版至少支持以下维度：

- `referrer_domain`
- `utm_source`
- `utm_medium`
- `utm_campaign`

归因基础版不做多触点归因模型，只做首层来源维度汇总统计。

### 11.8 收入规则

收入事件至少包含：

- 金额
- 币种

收入统计至少支持以下维度聚合：

- 日期
- 来源
- 币种
- 事件数
- 总收入

### 11.9 机器人过滤

系统应预留过滤常见机器人流量的规则位，但不要求实现复杂规则引擎。

## 12. 分析能力定义

### 12.1 概览

概览页至少展示：

- 页面浏览量 `PV`
- 访客数 `Visitors`
- 会话数 `Sessions`
- 自定义事件数
- 收入总额
- 时间范围趋势

### 12.2 页面分析

页面分析至少支持：

- 按页面路径统计浏览量
- 按时间范围筛选
- 热门页面排行

### 12.3 事件分析

事件分析至少支持：

- 按事件名称统计次数
- 查看事件排行
- 支持收入事件与普通事件共存

### 12.4 来源分析

来源分析至少支持：

- 按来源域名统计会话数
- 按来源域名统计收入

### 12.5 设备分析

设备分析至少支持：

- 浏览器维度
- 操作系统维度
- 设备类型维度

### 12.6 地域分析

地域分析至少支持：

- 国家
- 地区
- 城市

### 12.7 归因分析

归因基础版至少支持：

- `source / medium / campaign` 维度
- 会话数汇总
- 收入汇总

### 12.8 漏斗分析

漏斗基础版至少支持：

- 为某网站创建多个漏斗定义
- 每个漏斗由有序步骤组成
- 步骤类型至少支持：
  - 页面路径
  - 事件名称
- 查询结果至少返回：
  - 每步人数
  - 每步转化率
  - 每步流失量

### 12.9 留存分析

留存基础版至少支持：

- 按首访日期构建 cohort
- 统计 `D1`、`D7`、`D30` 留存
- 可按时间范围查询

### 12.10 收入分析

收入基础版至少支持：

- 收入总额
- 收入事件数
- 按来源维度汇总
- 按币种维度汇总

### 12.11 分析页差异化原则

各分析页不得仅复用相同表格后替换标题，必须具有明确差异化职责。

最少应满足以下要求：

- 概览页负责总览、趋势、关键入口与跨维度摘要，不承担完整明细职责
- 页面分析页聚焦内容路径、入口页、退出页与页面深度
- 事件分析页聚焦事件强度、事件类型分布、收入事件与会话覆盖
- 来源分析页聚焦来源结构、来源流量与来源收入
- 地域分析页聚焦国家、地区、城市三层分布
- 设备分析页聚焦浏览器、操作系统、设备类型与交叉矩阵
- 归因分析页聚焦 `source / medium / campaign` 结构与活动对收入的贡献
- 漏斗分析页聚焦步骤推进、转化率与流失
- 留存分析页聚焦 cohort 与 `D1 / D7 / D30` 热力观察
- 收入分析页聚焦收入总额、来源收入、币种结构与收入明细

实现约束：

- 每个分析页都应有自己的摘要区，而不是共用完全一致的卡片内容
- 每个分析页都应有自己的主表或主图，不允许全部页面退化为同一明细表
- 可共享表格组件与条形图组件，但不能共享相同页面语义

## 14. 核心实体规则

### 14.1 user

字段语义至少包括：

- 用户 ID
- 用户名
- 密码哈希
- 角色
- 是否启用
- 创建时间
- 更新时间

### 14.2 website

字段语义至少包括：

- 网站 ID
- 名称
- 域名
- 创建时间
- 更新时间

### 14.3 website_permission

- 用户 ID
- 网站 ID
- 访问级别 `access_level`
- `access_level` 仅允许：
  - `view`
  - `manage`
- 创建时间

- `website_permission` 表只存储已授权站点

### 14.5 share

- 分享 ID
- 所属网站 ID
- 公开 slug
- 是否启用
- 创建时间

### 14.6 visitor

字段语义至少包括：

- 访客 ID
- 所属网站 ID
- 外部标识
- 首次访问时间
- 最近访问时间

### 14.7 session

## 16. 页面行为要求

### 16.1 初始化页

- 未初始化时可访问
- 已初始化后不可重复创建管理员

### 16.2 登录页

- 支持用户名密码登录
- 登录成功后进入管理台

### 16.3 网站列表页

- 显示当前用户可访问的网站
- 支持创建网站
- 支持切换当前网站
- 若当前用户为 `super_admin`，可查看该网站的授权成员列表
- 授权成员列表用于核对范围，不承担主编辑职责

### 16.4 分析页面通用规则

所有分析页必须支持：

- 当前网站上下文
- 时间范围筛选
- 无权限时禁止访问

分析工作台还必须满足以下交互规则：

- 顶部全局上下文只保留当前页面真正需要的网站与时间范围控件
- 不得在所有页面无差别重复堆叠相同筛选控件
- 页面内局部筛选应服务于本页明细，例如页面路径筛选、事件名筛选、来源筛选
- 概览页中的关键模块应支持跳转到对应明细页
- 小屏幕下优先保证内容阅读，不得让顶部上下文条压缩主要内容区域

### 16.4.1 概览页

- 展示总体指标卡、趋势摘要、关键页面、关键来源、关键地域、关键设备
- 至少提供跳转到页面分析、来源分析、收入分析的入口
- 概览页不承担完整明细报表职责

### 16.4.2 页面分析页

- 展示页面表现摘要
- 展示热门页面、入口页、退出页
- 支持按页面路径筛选明细

### 16.4.3 事件分析页

- 展示事件表现摘要
- 展示重点事件、事件类型分布、收入相关事件信号
- 支持按事件名筛选明细

### 16.4.4 来源分析页

- 展示来源表现摘要
- 区分流量来源与收入来源
- 支持按来源名称筛选明细

### 16.4.5 地域分析页

- 必须分开展示国家、地区、城市三个层次
- 不允许只保留单一地域表格

### 16.4.6 设备分析页

- 必须分开展示浏览器、操作系统、设备类型
- 应提供设备组合或交叉矩阵视图

### 16.4.7 归因分析页

- 展示来源、媒介、活动三个层级
- 展示重点活动及其收入贡献
- 支持按归因关键词筛选明细

### 16.4.8 漏斗分析页

- 支持创建多个漏斗
- 支持查看每一步会话数、转化率、流失率
- 无管理权限时仅允许查看，不允许编辑定义

### 16.4.9 留存分析页

- 使用 cohort 形式展示留存
- 至少展示 `D1 / D7 / D30`
- 优先使用热力结构而不是普通列表

### 16.4.10 收入分析页

- 展示收入摘要、来源收入、收入明细
- 支持按来源或活动关键词筛选明细

### 16.5 用户管理页

- 仅 `super_admin` 可访问
- 支持创建、编辑、启停用户
- 支持配置站点授权
- 授权交互固定为 `none / view / manage`
- 页面内同时展示：
  - 用户列表
  - 角色说明
  - 用户编辑表单
  - 按网站分配权限
- 不单独提供“角色与授权”主页面

### 16.6 公开分享页

- 无需登录
- 只读展示
- 不暴露管理入口

### 16.7 设置页

- 展示系统运行信息、接入片段、公开分享入口与账户安全能力
- `super_admin` 可修改监听地址、数据库路径与日志等级
- 设置页不提供“默认语言”配置
- 系统语言默认跟随浏览器，用户可在前端界面内自行切换

## 19. 存储与查询原则

### 19.1 存储原则

- 使用 `SQLite`
- 同时存储原始事实数据与日聚合数据
- 聚合表用于支撑常用分析查询

### 19.2 查询原则

- 常用分析页优先读取聚合表
- 留存与漏斗可按需要读取原始事件 / 会话数据
- 查询必须受网站权限约束

### 19.3 时间粒度

基础聚合粒度固定为 `日`

### 19.4 性能取向

产品面向中小规模部署场景，不为超大规模高并发写入做承诺。

## 20. 前端与工程约束

### 20.1 前端技术约束

- 使用 `Vue 3`
- 使用 `<script setup>`
- 使用构建后的静态资源
- 不依赖 SSR

### 20.2 目录约束

- `server/`：Go 后端
- `server/embed/`：前端构建产物
- `web/`：前端源码
- `docs/`：产品与设计文档

### 20.3 发布约束

- 前端构建产物必须可嵌入后端
- 前后端版本必须随同一二进制交付

## 21. 国际化约束

- 界面至少支持中文和英文
- 文案资源集中管理
- 接口字段命名不随语言变化
- 数据模型与权限模型不依赖语言切换
- 浏览器语言是首次进入时的默认语言来源
- 若用户主动切换语言，可在前端本地持久化其选择

## 22. 安全与访问控制要求

- 密码必须以哈希形式存储
- 管理 API 必须要求认证
- 分享链接必须为只读能力
- 网站数据查询必须校验站点访问权限
- 关闭的分享链接必须不可访问
- 停用用户必须无法继续登录

## 23. 验收标准

当满足以下条件时，可认为产品达到交付要求：

1. 用户在空白环境中可通过单个二进制启动系统。
2. 首次启动无需手工建库建表。
3. 首次访问可完成管理员初始化。
4. 管理台、静态资源、采集接口、分享页由同一服务提供。
5. 用户可创建网站并获取接入代码。
6. 接入脚本或像素后，系统可接收到页面浏览与自定义事件。
7. 用户可在后台看到概览、页面、事件、来源、设备、地域、归因、漏斗、留存、收入等分析结果。
8. 用户权限与站点权限生效，未授权用户无法查看未分配站点数据。
9. 分享链接可公开访问但不能进行管理操作。
10. 系统不依赖 Node.js、PostgreSQL、Redis、ClickHouse 才能完成基础使用。
11. 同一代码库可构建 Linux、Windows、macOS 产物。
